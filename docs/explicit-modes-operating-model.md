# Explicit Modes — Your Operating Model

## The Insight

You don't work in continuous flow.

You work in **distinct cognitive modes**:

1. **Capture mode** — Input raw information without processing
2. **Design mode** — Build mental models and architecture
3. **Reconciliation mode** — Evaluate contradictions and updates
4. **Execution mode** — Act on decisions with minimal cognitive overhead

Each mode has:
- Different cognitive load
- Different rules
- Different governance needs
- Different friction levels

**The problem with continuous flow**: You're constantly switching between modes, which breaks your layered abstractions.

**The solution**: Make modes explicit. Transition deliberately. Apply governance at mode boundaries, not continuously.

---

## Part 1: The Four Modes

### Mode 1: Capture Mode

**What it is**: Collecting raw input without processing

**Cognitive load**: LOW (no analysis, no decision-making)

**What happens**:
- Inbox capture (Slack messages, emails, notes, ideas)
- Voice memos
- Screenshot collections
- Meeting notes (raw, unprocessed)
- Raw data input

**Governance**: MINIMAL
- Accept everything
- Don't judge it yet
- Don't try to organize it yet
- Just collect

**Rules**:
- ✅ Accept all input
- ✅ Capture with minimal processing
- ✅ Store with timestamp and source
- ❌ Don't analyze
- ❌ Don't decide
- ❌ Don't organize into structure

**Friction**: LOW (minimal gates; input flows freely)

**System requirements**:
- Quick capture (voice, notes, photos)
- Minimal structure (just store)
- Async (don't interrupt other modes)

**When you're in capture mode**: You're collecting. Don't design. Don't reconcile. Just capture.

---

### Mode 2: Design Mode

**What it is**: Building mental models, architecture, strategy

**Cognitive load**: VERY HIGH (requires all 4 abstraction layers)

**What happens**:
- System architecture design
- Strategic thinking
- Problem formulation
- Creating mental models
- "What should we build?" thinking

**Governance**: MINIMAL (during execution)
- Protect focus
- Protect abstraction layers
- No interrupts
- Applied at mode ENTRY/EXIT, not during

**Rules**:
- ✅ One task at a time
- ✅ Uninterrupted focus
- ✅ Phone/notifications off
- ✅ No meetings scheduled
- ❌ No context switches
- ❌ No interrupts

**Friction**: HIGH (controlled entry; strict exit)
- **Entry**: Simple (you declare you're in design mode)
- **Execution**: Frictionless (no governance during; pure focus)
- **Exit**: Simple (you finish or hit 10am boundary)

**System requirements**:
- Protected time block (6-10am)
- Calendar blocking
- Notification silencing
- Accountability at boundaries only (not during)

**When you're in design mode**: Layers stabilize. Deep thinking happens. This is your superpower. Protect it.

---

### Mode 3: Reconciliation Mode

**What it is**: Evaluating contradictions, updating beliefs, processing conflicts

**Cognitive load**: HIGH (requires analytical thinking + emotional processing)

**What happens**:
- Compare new info against existing knowledge
- Identify contradictions
- Update mental models
- Re-evaluate decisions
- "What changed?" thinking

**Governance**: MODERATE (structured process)
- Compare: Is this new input consistent with existing models?
- If consistent: Accept and integrate
- If contradictory: Flag and evaluate
- If novel: Add as new model branch

**Rules**:
- ✅ Structured comparison process
- ✅ Evidence-based evaluation
- ✅ Document contradictions
- ✅ Update models explicitly
- ❌ Don't leave contradictions unresolved
- ❌ Don't silently drop old models

**Friction**: MODERATE (gated entry; structured process)
- **Entry**: Triggered by captured items + design review
- **Execution**: Follow reconciliation process
- **Exit**: Documented updates to models

**System requirements**:
- Structured reconciliation process (compare/evaluate/update)
- Contradiction log
- Model versioning
- Explicit update tracking

**When you're in reconciliation mode**: You're updating your beliefs. This is where growth happens, but it's expensive. Use deliberately.

---

### Mode 4: Execution Mode

**What it is**: Taking action on decided design

**Cognitive load**: LOW-MEDIUM (less thinking; more doing)

**What happens**:
- Implementation
- Shipping
- Operations
- "Do the work" thinking
- Execution is low-friction once design is decided

**Governance**: MINIMAL (once direction is set)
- Direction already decided (in design mode)
- Just execute
- Governance applied at mode boundaries (did design capture all needs?)
- During execution: minimal overhead (trust the design)

**Rules**:
- ✅ Follow the design
- ✅ Ask for clarification at boundaries
- ✓ Surface blockers early
- ❌ Don't re-question the design mid-sprint
- ❌ Don't start new design mid-execution

**Friction**: LOW (once you enter; minimal stopping)
- **Entry**: Design is complete and documented
- **Execution**: Pure action, minimal governance
- **Exit**: Deployed, running, or handed off

**System requirements**:
- Clear design documentation (from design mode)
- Execution plan
- Clarity on success criteria
- Blockers escalation path (not constant re-approval)

**When you're in execution mode**: You're shipping. Don't second-guess the design. Trust it and execute.

## Part 2: The Mode Transitions

The **transitions** are where governance happens, not during modes.

### Capture → Design Transition

```
Capture mode ends → Items collected
  ↓
[TRANSITION POINT — Governance applied here]
  ↓
Question: Which captured items warrant design thinking?
  ↓
Sort: What's action item, what's noise, what's signal?
  ↓
Plan: How much design time do we allocate?
  ↓
Design mode begins with clear scope
```

**Low-friction execution principle**: 
- During capture: Accept everything (no judgment)
- At transition: Curate ruthlessly (high judgment)
- During design: Execute on curated items (no filtering)

### Design → Reconciliation Transition

```
Design mode produces: New mental model
  ↓
[TRANSITION POINT — Governance applied here]
  ↓
Question: Does this new design contradict existing models?
  ↓
Compare: Against current mental models, prior designs, team decisions
  ↓
Reconcile: Update models or document contradiction
  ↓
Execution mode begins with updated models
```

**Low-friction execution principle**:
- During design: Pure creativity (no judgment)
- At transition: Compare and reconcile (high judgment)
- During execution: Execute on reconciled design (no questioning)

### Design → Execution Transition

```
Design mode ends → Design documentation complete
  ↓
[TRANSITION POINT — Governance applied here]
  ↓
Questions:
- Is design clear enough to execute?
- Are dependencies documented?
- Are edge cases covered?
- Are success criteria defined?
  ↓
If yes → Execution mode begins
If no → Back to design mode
```

**Low-friction execution principle**:
- During design: Design thoroughly (high friction at boundaries)
- At transition: Gate thoroughly (ensure clarity)
- During execution: Trust and act (low friction during action)

### Execution → Capture Transition

```
Execution produces: Results, blockers, learnings, new questions
  ↓
[TRANSITION POINT — Governance applied here]
  ↓
Question: What did we learn?
  ↓
Capture: Blockers, unexpected patterns, learnings
  ↓
Design: New questions, refinements, adaptations
  ↓
Cycle repeats
```

## Part 5: Rules for Each Mode

### Design Mode Rules

1. **Protected time**: 6-10am, non-negotiable
2. **One task**: No context switches
3. **Layer stability**: Phone off, notifications off, no interrupts
4. **Governance at exit**: Before leaving design mode, document:
   - What was designed?
   - What assumptions were made?
   - What needs reconciliation?
5. **Time-box**: When 10am hits, transition (don't leak into other time)

### Capture Mode Rules

1. **Accept all**: Don't judge while capturing
2. **Timestamp all**: When was it captured?
3. **Source all**: Where did it come from?
4. **Minimal processing**: Just store; don't organize
5. **Triage at boundaries**: When exiting capture, sort by urgency

### Reconciliation Mode Rules

1. **Structured process**: Compare → Evaluate → Decide → Update
2. **Evidence-based**: Use data, not feelings
3. **Document contradictions**: Don't ignore; flag for design
4. **Update models explicitly**: Version your mental models
5. **Don't stay in reconciliation**: Once decided, move to design or execution

### Execution Mode Rules

1. **Trust the design**: Don't re-question during execution
2. **Follow the plan**: Execute as designed
3. **Escalate blockers**: Don't solve; surface early
4. **Don't redesign mid-sprint**: Store new ideas for next design cycle
5. **Ship and learn**: Execution produces learning; capture it

## Part 6: What This Fixes

### Your Original Problems

**"Trouble staying on task"**
- ✓ Fixed by: Explicit design mode with protection (no context switches)
- Your abstraction layers can stabilize because there's no switching

**"Easily distracted"**
- ✓ Fixed by: Capture mode is separate from design mode
- Distraction happens in capture; design is protected
- You're not switching between modes; you're in one mode at a time

**"Too much to focus on"**
- ✓ Fixed by: Visible system state shows pending, trusted, needs reconciliation
- You can see priorities clearly; not everything needs your attention now

**"Can't focus until meetings are over"**
- ✓ Fixed by: Meetings are mode transitions, not interrupts
- Meeting happens at planned boundary (10am = end of design mode)
- Recovery happens as part of transition to next mode

**"Energy crashes by 3pm"**
- ✓ Fixed by: Layers don't need rebuilding if you're in execution mode (lower cognitive load)
- Design mode ends before crash; execution is lower cost

## Summary

Your operating model isn't continuous flow. It's **explicit modes with low-friction execution during modes and governance at transitions**.

```
Design Mode (6-10am)  → Protected, high focus, builds layers
     ↓ [TRANSITION: Document design, flag reconciliation items]
Capture Mode (10am-12pm) → Low friction, accept all, minimal judgment
     ↓ [TRANSITION: Triage urgent items, decide next mode]
Execution Mode (1-3pm) → Trust design, ship, surface blockers
     ↓ [TRANSITION: Collect learnings, flag contradictions]
Reconciliation Mode (as needed) → Compare, evaluate, update models
     ↓ [TRANSITION: Document decisions, return to design or execution]
```

This is how you actually work. Build your systems around it.
